# CLAUDE.md — sparkshark-com

## Repo Purpose
Public marketing website for **Spark Shark Electric** — sparkshark.com.

A static HTML site with **no frameworks, no JS bundlers, no Node toolchain**. Every HTML file in the repo is generated by `build.py` (Python stdlib only) from a single in-script manifest. Output is committed alongside the generator. Two static hosts serve from `main`: Vercel rebuilds with `BASE=""` at deploy time via `vercel.json` `buildCommand`, and the transitional github.io preview serves committed HTML directly with `BASE=/sparkshark-com`. Both produce the right asset paths for their respective domains.

Pages covered: homepage, ~15 service pages, ~9 city/location pages, blog posts, plus standard footer pages (about, contact, reviews, FAQ, privacy, terms, 404). Every page emits a 4-node JSON-LD `@graph` (WebSite + Organization + LocalBusiness/Electrician + FAQPage); service pages add BreadcrumbList + Service; blogs add BreadcrumbList + Article.

## Business Context
This is the primary lead-generation surface for Spark Shark Electric (residential electrical, OKC metro). It carries the NAP, license, BBB info, ServiceTitan Scheduler Pro embed (online booking), and all SEO/GEO content. Anything that breaks schema, NAP consistency, or the scheduler embed directly hurts lead flow.

## Quality Gate (`qa.py`)

Pure-Python content-quality checker that runs after every build. Verifies each generated HTML page against an 8-rule registry: scaffolding leaks (`(centered, large)` in headings, `=== SECTION:` markers), unfilled `[VERIFY: …]` placeholders, orphan label bullets, stale strings (Flanco Electric, prior phone numbers, prior addresses), brand misspellings (`SparkShark`/`Sparkshark` flagged), NAP presence (phone / city / ZIP / license on every page), license placement (`#163603` only inside `<footer>`; JSON-LD ignored), and required marketing assets (mascot + logo `<img>` on primary marketing pages).

```bash
# After editing copy / build.py
python3 build.py                 # rebuild the HTML
python3 qa.py                    # full sweep — every sitemap page + /404.html
python3 qa.py /reviews/          # check specific page(s)
python3 qa.py --ci               # machine-readable for CI logs
python3 qa.py --strict           # promote warnings (e.g., `owner`) to failures
bash tests/test-detection.sh     # canary regression test (must PASS)
```

`qa.py` reads `sitemap.xml` for the URL manifest and writes `qa-report.md` at repo root. Sitemap entries with no on-disk file (e.g., zombie dirs from de-listed pages) are skipped with a stderr warning, not failed — sitemap consistency is `build.py`'s responsibility.

**CI gate:** `.github/workflows/qa-on-pr.yml` runs `build → qa --ci → tests/test-detection.sh` on every PR + push to `main`. Any FAIL blocks merge. The full `qa-report.md` uploads as a workflow artifact (30-day retention). On failure, a structured comment is auto-posted to the PR with the per-page breakdown.

**When CI is red:**
1. Open the PR; the auto-posted "QA gate failed" comment has the per-page breakdown inline.
2. If a real content bug fired, fix the source in `copy-drafts/*.md` (or `build.py`), run `python3 build.py`, commit + push. CI re-runs automatically.
3. If `qa.py` is mis-firing on legit copy, mention `@claude` in a PR comment (Claude Code GitHub App) or run `subscribe_pr_activity` from Claude Code on the web — the next Claude session can investigate using this CLAUDE.md as context.

**Adding rules:** Edit the registry at the top of `qa.py` — `STALE_STRINGS`, `SCAFFOLDING_PATTERNS`, `BRAND_MISSPELLINGS`, `DISCOURAGED_WORDS`, `NAP_REQUIREMENTS`, `MARKETING_PAGES`, `PLACEHOLDER_PATTERNS`. Adding a new stale string is a one-line list append. Brand / NAP values must stay in sync with `BRAND` in `build.py` (the dependency is documented in `qa.py`'s docstring).

**Canary regression guard:** `tests/test-detection.sh` injects a known scaffolding bug (`<h2>RATING SUMMARY (centered, large)</h2>` + orphan `Stars:` / `Number:` bullets) into `reviews/index.html`, asserts `qa.py` catches it and names "centered, large" in the report, then restores the file. Runs after `qa.py --ci` in CI. If anyone weakens `qa.py` such that the canary stops firing, the PR is blocked.

### copy-drafts structure
`copy-drafts/` is the **human-editable layer** for site copy. Each file is one page, named with a 2-digit prefix that controls reading order:

```text
01-homepage.md
02-electrical-panels.md … 15-residential-electrical-solutions.md   # service pages
16-services-index.md
17-about-us.md     18-contact-us.md     19-reviews.md     20-faq.md
21-privacy-policy.md     22-terms-and-condition.md
23-locations-index.md
24-oklahoma-city.md … 33-newcastle.md                              # city pages
34-blog-power-out.md … 37-blog-generator-installation.md           # blog posts
38-blogs-index.md
```

Per-file format (enforced by `extract-copy-drafts.py`):
```text
# TITLE (browser tab)
# META DESCRIPTION (Google snippet)
# H1
# HERO SUBTITLE
# INTRO
# SECTIONS (h2 + bullets)
# FAQ (Q/A pairs)
```

**Workflow:** edit the `.md` files → merge changes back into `build.py`'s manifest → re-run `python3 build.py` → commit both the build.py change and the regenerated HTML in the same commit.

## Guardrails
- Inspect `build.py` before editing any HTML — most HTML changes belong in `build.py` (or `copy-drafts/` → `build.py`), not in the generated `index.html` files. Hand-edits to generated HTML will be overwritten on the next `python3 build.py`.
- State exact files before modifying.
- Do not refactor unrelated areas (no Jekyll, no Hugo, no Node, no React — the "no frameworks" stance is intentional; both Vercel and the github.io preview serve raw static output, no framework runtime needed).
- Do not add frameworks or dependencies without approval.
- Do not commit/push/deploy unless Brock approves. (Vercel and the github.io preview both deploy on every push to `main`, so a push IS a deploy. Once DNS flips to Vercel — gated by `docs/migration/launch-gate.md` — pushes go live to www.sparkshark.com immediately.)
- Preserve the 4-node `@graph` schema architecture and NAP consistency on every page.
- Spark Shark claim rules from global CLAUDE.md apply: do not invent reviews, ratings, BBB status, license info, response times, or testimonials. The numbers in `BRAND` (`build.py`) are the single source of truth — if they need to change, change them there and regenerate.

## Acceptance Checks
A change is done when:
```text
1. python3 build.py runs cleanly with no Python errors.
   (And BASE="" python3 build.py also runs cleanly if production paths matter
   for this change.)

2. `git status` shows the regenerated HTML matches what was intended — no
   stray files, no half-regenerated pages, no leftover preview-mode asset paths
   if the change is destined for production.

3. `git diff` on a sample generated page (e.g. index.html and one service page)
   reflects the intended copy/markup change and nothing else.

4. JSON-LD on at least one affected page validates as expected (4-node @graph
   present; BreadcrumbList + Service on service pages; Article on blog posts).
   Quick visual check: search the page source for `"@graph"` and confirm the
   four node @types are still WebSite, Organization, Electrician, FAQPage.

5. Local smoke test: `python3 -m http.server 8000`, open the changed page(s),
   confirm: header/footer render, phone link is `tel:+14054364776`, mobile
   menu opens, ServiceTitan scheduler embed loads (homepage + contact),
   internal nav links resolve (no 404s under the BASE used).

6. NAP unchanged unless the task explicitly changes it: phone (405) 436-4776,
   address Moore OK 73160, license #163603.

7. After push to main, wait for deploys:
   - Vercel preview/production builds publish in ~10–30s via Git integration
     (running `BASE="" python3 build.py` per vercel.json).
   - GitHub Pages preview also republishes in ~30s during the transitional
     window (built with default BASE).
   Verify the change on whichever URL is the right one for the work
   (Vercel preview, github.io preview, or live www.sparkshark.com after
   cutover). Reminder: do not flip DNS to Vercel until every gate item in
   `docs/migration/launch-gate.md` is Approved or Not Applicable.
```

## Rollback
```text
Single bad commit, not yet built upon:
    git revert <sha> && git push origin main
    # Vercel republishes the prior state on the next deploy (~10–30s).
    # GitHub Pages preview also republishes in ~30s.

Multiple bad commits in a row, recovering to a known-good SHA:
    git revert <oldest_bad_sha>^..<newest_bad_sha> && git push origin main
    # Creates a clean revert commit chain. Do NOT force-push main.

Broken build.py output (HTML went sideways):
    git checkout <good_sha> -- <path/to/file_or_dir>
    git commit -m "rollback: restore <area> from <good_sha>"
    git push origin main
    # Vercel rebuilds via buildCommand; the bad output is gone next deploy.

Wrong BASE baked into a build (most likely now: someone hand-edited generated
HTML with /sparkshark-com paths, or someone reset vercel.json buildCommand):
    # Restore vercel.json or fix the offending HTML, then push.
    # Vercel always rebuilds with BASE="" via buildCommand; CI continues to
    # rebuild with default BASE for the github.io preview.

DNS rollback (POST-CUTOVER ONLY):
    # If Vercel cutover fails, revert DNS to pre-cutover values at GoDaddy.
    # This is why docs/migration/launch-gate.md item #5 (current DNS records
    # screenshot) is required BEFORE cutover — without it, rollback is blind.
    # Pre-cutover, rollback is moot: there is nothing to roll back.

Never rewrite history on main (no force-push, no rebase of pushed commits).
```
