### A. WordPress decommission risks specific to a "rebrand-in-place" install

The WP install is still named `flancoelectric` and has been running continuously since the Flanco era through multiple agency handoffs. What surprises does an in-place-rebranded WordPress install typically hide that could fire during or after cutover and damage the new static site?

Examples of the kind of thing to surface:

- Plugin-side webhooks that POST to old endpoints during the cutover window
- `wp_options` entries with hardcoded URLs pointing to flancoelectric.com or flancoelectricok.com
- Scheduled WP-cron jobs that trigger third-party syncs (CRM, ESP, social-media auto-poster, review-aggregator) under the old brand
- Site Kit / Google Site Kit / Monster Insights / Analytify installs left from the Flanco era still siphoning data into a stale GA property
- Yoast / RankMath / All-in-One SEO sitemap caches still listing old Flanco-era slugs after the static cutover
- `mu-plugins/` directory contents left by prior agencies (Coalition, Conversionly, Strategy Zoo, Ad.IQ, etc.)
- `functions.php` hooks in the active theme that emit Flanco strings on canonical-URL changes
- WP Mail SMTP / Post SMTP / FluentSMTP plugins still authenticated against an old mailbox
- Transient API tokens cached in `wp_options` for services we've offboarded
- Stale WAF rules in WP Engine that conflict with Vercel's edge headers
- WP-CLI scheduled tasks, custom Action Scheduler queues
- Wordfence / Sucuri / iThemes-Security old IP allowlists referencing prior agency IPs
- WP Engine-specific systems: GeoTarget, Genesis blocks, Smart Search, EverCache rules

### C. Common cutover failure modes for WP → static migrations specifically in service businesses

What goes wrong that does not appear in our checklist? Cite case studies, post-mortem write-ups, vendor docs, or community threads.

Examples of concerns:

- DNS propagation gotchas where some visitors see the new site and some see the old for 24–72 hours, and what's the right lead-capture posture during the split window.
- SSL/TLS cert issuance edge cases on Vercel (Let's Encrypt). What happens to CAA records, what if there's a stale cert on WP Engine, what if there's a HSTS preload entry from a prior brand?
- Email DKIM / SPF / DMARC: we use Resend for transactional + Google Workspace for inbox at @sparkshark.com. If WP Engine's SPF includes any record referencing the old mail path, does it break Resend deliverability post-cutover?
- WordPress's `xmlrpc.php`, `wp-json/`, `wp-cron.php`, `wp-login.php` — are they being scraped/probed in ways that could create false-positive 404 floods or alert-fatigue post-cutover?
- Old WP redirects buried in `.htaccess`, `web.config`, NGINX, or WP Engine-specific config layers that we don't see because WP Engine's UI handles them differently.
- Schema markup ownership transition: does Google's understanding of the entity (LocalBusiness, sameAs links, knowledge graph cluster) survive a host change cleanly when the install name internally is still tied to a defunct brand?
- ServiceTitan Scheduler Pro embed: does it carry CSP / CORS / referrer requirements that might fire only on the live domain (not the preview alias)?
- Google's freshness signals: does a host change trigger a temporary re-crawl that exposes thin / orphan pages we haven't surfaced?
