### K6 — Hotlinked media (HIGH — but resolves on cutover)

Live WP currently hotlinks **7 image references to `i0.wp.com`** (Jetpack Photon CDN — Automattic's WP.com image proxy). Examples:
- `https://i0.wp.com/www.sparkshark.com/wp-content/uploads/2023/07/favicon-1.png?fit=32%2C32&ssl=1`
- `https://i0.wp.com/www.sparkshark.com/wp-content/uploads/2023/07/favicon-1.png?fit=192%2C192&ssl=1`
- … and 5 more image references.

**Net finding:** This is a Jetpack hotlink pattern (the WP site routes its own images through the Photon CDN for caching). When the WP install is decommissioned or Jetpack is deactivated post-cutover, **every Photon URL goes dark** — they cannot be reproduced from the new build. The new build serves its own images out of `img/` under root-relative URLs, so the *new* site is unaffected. But anything that has these URLs cached (search-engine image indexes, AI assistant context, archive crawls) will see broken images post-cutover.

**Action:** for the assets referenced by these 7 Photon URLs that survive on the new site, no action needed — the new build replaces them with self-hosted versions. For any image the new build does NOT replace, save a copy from Photon before WP is decommissioned. The favicon (next section) is the most exposed asset.

### K8 — Favicon hygiene (HIGH — pre-cutover-fixable)

**Live WP:** dedicated 32×32 / 192×192 / 180×180 favicon at `/wp-content/uploads/2023/07/favicon-1.png` (proxied through `i0.wp.com`). Probed at 32×32 → HTTP 200.

**New Vercel build:** `build.py` line 365–366:
```html
<link rel="icon" type="image/png" href="/img/logo.png">
<link rel="apple-touch-icon" href="/img/logo.png">
```
The favicon and the Apple touch icon **both reuse `logo.png`** — the full brand logo file, not a dedicated favicon asset. Two issues:

1. **No dedicated favicon asset in the new build.** `logo.png` at favicon size (32×32) will render as a downscaled version of the full Spark Shark Electric logo — unreadable mascot/wordmark at tab-bar size.
2. **No 192×192 PWA icon, no 180×180 Apple touch icon, no `manifest.json`.** The current build serves one URL three roles.

**Cache aggression risk:** browsers aggressively cache favicons; once Google's preview crawler caches the downscaled-logo favicon, replacing it later requires the new asset to live at a different URL or with a different `?v=` query string. Best-case fix is to ship the dedicated favicon set **before** DNS flips, so the very first crawl serves the right thing.

**Recommended:** generate three assets — `img/favicon-32.png`, `img/favicon-192.png`, `img/apple-touch-icon-180.png` — and update `build.py` lines 365–366 to reference them by size. Optionally add `img/site.webmanifest` for PWA/install-prompt support. One-file PR.
